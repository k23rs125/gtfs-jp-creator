# コスモス号（北野地域よりみちバス）で判明した座標方式の課題と改良方針

対象スクリプト: `scripts/extract_timetable_coords.py`（Step1 第三経路＝座標方式）
発見日のテストPDF: `kitanoB.pdf`（コスモス号 B日程、令和6年6月改正版）/ `R7kitanoA.pdf`（A日程、令和7年4月改正版）
目的: 未知PDFでスキルの適用範囲・弱点を洗い出す。本メモは次回の改良着手用の設計メモ。

## 0. 要約（次に直すこと）

1. **`--blocks` 手動指定が実装されていない**（ヘルプと実装の食い違い＝バグ）。
2. **ブロック分割が「停留所名列の x 座標の塊」だけに依存**しており、名前列 x が近接する
   複数方面（弓削線の2方面）を分離できない。別方面の便を1ブロックに同居させ、
   時刻の異常な逆行を生む。
3. （A日程 page2 限定）**停留所名が画像化**されており座標方式は適用外。時刻数字のみ
   テキスト。これは「正しく失敗」で入手案内に回すべきケース。

## 1. テストPDFの素性（判定済み）

`len(page.chars)` と日本語/数字トークン数で判定（基準: chars<20 で画像PDF）。

| ファイル | ページ | chars | 日本語 | 数字 | 判定 |
|---|---|---|---|---|---|
| R7kitanoA.pdf | page1(経路図) | 18 | - | - | 画像 |
| R7kitanoA.pdf | page2(時刻表) | 1454 | **21** | 1402 | 時刻はテキストだが**停留所名が画像化**→座標方式 適用外 |
| kitanoB.pdf | page1(経路図) | 1990 | - | - | 文字あり |
| kitanoB.pdf | page2(時刻表) | 5750 | **1621** | 3416 | 停留所名もテキスト→座標方式が効く |

結論: **B日程(kitanoB.pdf) page2 のみ座標方式の対象**。A日程は停留所名が画像化されており、
資料の方針どおり「テキスト版PDF/元データ(Excel)の入手を促す＝正しく失敗」が正解。
同一事業者・同一路線でも改正版によりPDFの素性が変わりうる、という重要な知見。

## 2. 抽出の素の性能（うまくいっている点）

`python scripts/extract_timetable_coords.py kitanoB.pdf -o out.json --page 2`（自動・2ブロック）

- 停留所名・番号(num)・時刻・要予約(reserve)を、循環・往復込みで seq 順に正しく抽出。
  例: 北野線循環で「古賀茶屋駅(8:24)→…→陣屋第一→…→古賀茶屋駅(9:28)」を順序通り。
- warnings は空。怪しいものは needs_confirmation に隔離（「正しく失敗」が機能）。
- 検出: block0(name_x=137.5) 停留所71 便6 / block1(name_x=816.5) 停留所54 便6。

## 3. 不具合と原因（コード行番号付き・確定）

### 不具合A: `--blocks` が無視される（バグ）
- 引数定義は 197-198 行。しかし `args.blocks` を参照する箇所がコード中に存在しない。
- ブロック数は 245 行 `block_centers = cluster_cols(name_x0s, 50)` だけで決まる。
- 実害: `--blocks 3` を渡しても 2 ブロックのまま（出力・needs件数とも変化なし）で再現。

### 不具合B: 名前列xの塊だけでブロックを割るため複数方面を分離できない（設計限界）
- 243 行: 出現10件以上の x0 を名前列候補 `name_x0s`。
- 245 行: それを gap=50px でクラスタした `block_centers` がブロック中心。
- 254-258 行: 各ブロックの時刻列範囲を
  `time_x_lo = cx+120`, `time_x_hi = 次の名前列x - 60`（無ければページ右端）で決める。
- 問題: 弓削線の2方面（生涯学習センター方面と古賀茶屋駅方面）は**名前列xが近接**して
  1つ(816.5)に統合される。すると 258 行により、その1名前列の右側にある
  **2方面ぶんの時刻列(col_x≈986〜1360)が全部1ブロックに入る**。
- 結果: 別方面の便を同じブロックの便として拾い、循環で同名停留所が複数回出ることと
  あいまって、便内で時刻が大きく逆行する（実測: block1 col_x=1209.5 で
  seq23 14:49 → seq24 13:35 と約1時間の逆行）。これは寄り道では説明できない誤連結。

### 副次C: 時刻表外の案内文を停留所候補に拾う
- block1 末尾で「お/問/い/合/わ/せ【事業主体】」等（“お問い合わせ”“事業主体”の縦割れ）を
  停留所候補として検出。ただし番号が取れないため uncertain→needs_confirmation に隔離。
- 「正しく失敗」は機能しているが、時刻表の下端より下（案内文領域）を行範囲から除外すれば
  より綺麗になる。

## 4. 改良案の評価（次回の着手方針）

- **案1: `--blocks` を実際に効かせる。** 245 行で、クラスタ結果が指定数と異なるとき gap を
  調整して指定数に合わせる。手軽。ただし弓削線2方面のように名前列xが実質1つしか
  立たないケースでは、これだけでは3に割れない。→ 部分的。

- **案2（推奨）: 便の時刻列(col_x)の大きな空白でもブロックを割る。** 名前列xが同じでも、
  時刻列に大きなギャップ（生涯学習センター方面 col_x≈986〜1133 と
  古賀茶屋駅方面 col_x≈1209〜1360 の間の空白）があれば、そこを境にブロックを分割する。
  254-258 行のブロック生成を「名前列単位」から「名前列×時刻列ギャップ」へ拡張。
  コスモス号B日程を正しく3方面（北野線/弓削線2方面）に分離できる見込み。本質的。
  実装は中核に触れるため影響大。

  **【実測による訂正】案2は弓削線2方面には効かないと判明。** 実データで時刻列の間隔を
  測ると、北野線(〜675)と弓削線(986〜)の境は311pxの大ギャップで割れるが、弓削線内部の
  2方面の境（1133→1209）は74pxで、通常の便間隔（73〜77px）と区別できない。名前列xも
  共有（弓削2方面とも x≈816）。よって**座標（名前列x・時刻列ギャップ）では弓削2方面を
  分離できない**ことが確定。案2は採用見送り。混在の「検出と明示」（案3寄り）に切替＝改良X。

- **案3: 後段で1便内の大きな時刻逆行を検出し別便に切る。** 対症療法で誤連結の根本は
  直らない。案2の補助的な検査としてなら有用（分割後の健全性チェック）。

**推奨**: 案2を主、案1（`--blocks` を効かせて手動オーバーライド可能に）を補助、
案3を分割後の検証として併用。

## 5. テスト時の必須確認（リグレッション防止）

- 中核（ブロック分割）を変更するため、**インガット号で確立した誤差ゼロを壊さないこと**を
  必ず確認する。改良後、インガット号A/B日程で従来と同じ抽出結果（停留所・便・時刻が
  一致）になるかを回帰テストする。
- コスモス号B日程は、改良後に「北野線循環 / 弓削線・生涯学習センター方面 /
  弓削線・古賀茶屋駅方面」の3ブロックに分かれ、各便内の異常な時刻逆行
  （seq境界での1時間級の戻り）が消えることを成功条件とする。
- 要予約バス停への寄り道による数分程度の逆行は「正当な逆行」。これは消さない
  （needs_confirmation での提示は維持）。異常（列の誤連結）と正当を混同しないこと。

## 6. 再現手順

```
# 素性判定
python - <<'PY'
import pdfplumber
for p,pg in [("R7kitanoA.pdf",1),("kitanoB.pdf",1)]:
    with pdfplumber.open(p) as pdf:
        page=pdf.pages[pg]; ch=page.chars
        jp=[c for c in ch if any('\u3040'<=x<='\u30ff' or '\u4e00'<=x<='\u9fff' for x in c['text'])]
        print(p,"page",pg+1,"chars",len(ch),"jp",len(jp))
PY

# 抽出（自動=2ブロックになる。--blocks 3 を渡しても現状は無視され2のまま）
```

## 7. ステータス

### 改良X 実装済み（コミット 92c8b98）
本メモ section 4 の「案1（補助）＋案3（検証）」に相当する改良Xを実装・コミット済み。
`scripts/extract_timetable_coords.py` に以下を追加（既存ロジックは温存）：

1. **`--blocks` 手動指定を有効化**（不具合Aの修正）。`--blocks N` 指定時、名前列クラスタの
   gap を調整して N に近づける。便を持つ最終ブロック数が N に満たない場合は warnings に
   「座標からは M ブロックしか分離できません。原典で方面ごとに確認」と明示（正しく失敗）。
2. **複数方面混在の検出と警告**（不具合B への現実的対応）。便内で30分以上の時刻逆行が
   あれば needs_confirmation に `multi_pattern_suspected` を追加し、対象列と戻り幅を提示。
   「1つの名前列に複数方面/パターンの便が同居している可能性。原典で方面ごとに確認」と促す。
   閾値は `--multi-pattern-min`（既定30分）で調整可。

**閾値30分の根拠（実測）**: インガット号A/Bの正当な循環の戻りは最大8分。コスモス号の弓削線
2方面混在は74分・56分。30分なら誤検出なく混在のみ捕捉できる（8分と56分の間に十分な余裕）。

**回帰テスト結果**: インガット号A/B日程の抽出結果が改良前とバイト単位で完全一致
（jojimaA SHA256 46626051… / jojimaB SHA256 93d5e199…）。誤差ゼロを維持。
コスモス号B日程では multi_pattern_suspected が block1（弓削線、列1209.5=74分・1283.1=56分）で
発火、`--blocks 3` 指定時は「2ブロックしか分離できない」警告が出ることを確認。

### 未対応（次の課題）
- **弓削線2方面の本質的な分離は未対応**（案2は座標的に不可能と確定したため見送り）。
  生涯学習センター方面と古賀茶屋駅方面は、名前列x も時刻列の間隔（共に約74px）も
  通常の便間隔と区別できず、座標だけでは分割不能。現状は「混在を検出して人に明示」
  までが座標方式の限界。本質的分離には、行先見出しの座標利用や便の起終点パターン解析など
  座標方式の枠を超えるアプローチ（別途設計）が必要。これは別タスクとして残す。
