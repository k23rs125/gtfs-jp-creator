# 座標精度のゴールデンテスト — 知見レポート

**対象**: バス時刻表 → GTFS-JP 半自動生成システムの**停留所座標の正確性**
**方法**: 公式GTFS（BODIK）を"正解"に、本システムの独自手法（公式再利用OFF）を突き合わせる回帰テスト
**再現**: `python golden_test.py`（比較の実体は `skills/gtfs-jp-creator/scripts/eval_compare.py`）
**本レポートの主眼**: 「精度が完璧か」ではなく **「黙って誤りを出していないか（正しく失敗しているか）」** を定量化する。

---

## 1. 方法（ゴールデンテスト）

公式GTFSが存在する路線について、本パイプラインを **公式座標の再利用OFF**（＝独自手法 P11＋Nominatim＋経路内挿のみ）で実行し、公式GTFSを"正解"として比較する。

- **対象路線**: 柳川市コミュニティバス「両開にしルート」（時刻表PDFから生成、停留所35）
- **正解データ**: 柳川市公式GTFS（BODIK, 全路線173停留所）
- **一致付け**: 停留所名を NFKC＋全角/半角空白正規化して照合。名前一致の停留所どうしで座標の距離（Haversine, m）を測る。
- **信頼度**: 各停留所を `確定 / 要確認 / 未補完` に自動分類（`classify_coord_confidence`）。

> 公式座標を再利用すると当然ながら座標は一致（誤差0m）するため、**独自手法の実力を測るには公式再利用をOFF**にして比較する必要がある（本テストの肝）。

---

## 2. 結果

### 2.1 座標の生成方法（source）別の正確さ

名前一致の停留所について、当方座標と公式座標の距離（中央値）を source 別に集計。

| 生成方法 | 停留所数 | 距離の中央値 | 備考 |
|---|---:|---:|---|
| P11 完全一致（経路整合） | 2 | **57 m** | 正確 |
| Nominatim（OSM）補完 | 9 | **47 m** | 正確（ただし2/9は>100m） |
| P11 あいまい一致 | 1 | 225 m | 要確認相当 |
| **経路内挿（推定）** | **23** | **405 m** | データが無い停留所の穴埋め |

- 独自手法だけでの**全体の中央値は約 232 m**（Nominatim ON）。**Nominatimを切ると約 2,000 m** に悪化する（＝Nominatimの寄与が大きい）。

### 2.2 なぜ内挿が多いのか（データの壁）

- **P11（国土数値情報 バス停, 福岡県 8,490停留所）に、柳川コミュニティバスの停留所がほぼ無い。**
  「六十丁」「中ノ切」「西ノ切」「両開」「下宮永」「うぶすな」等は **P11 に 0 件**。
- P11 の"一致"の一部は**同名の別地点**（例: 「外開」→十数km先の別の外開、「明治」→「明治町」）で、**経路外れ値として棄却**され内挿に降格された（＝誤マッチを弾いている）。
- Nominatim（OSM）も、これら超ローカル名は**地物として存在せず geocode 不能**（35中21件が失敗）。
- 結果、**信頼できる座標が付くのは 35 中 約11件**にとどまり、残り 23 件は内挿で埋めるしかない。

### 2.3 信頼度 × 正確さ（＝本研究の核心指標）

「≤100m を正確」とみなし、信頼度とクロス集計した（公式と名前一致・両方に座標がある停留所）。

| 信頼度 | ≤100m（正確） | >100m（不正確） |
|---|---:|---:|
| **確定** | **2** | **0** |
| 要確認 | 0 | 26 |
| 未補完 | 0 | 0 |

- **「確定」は 2/2 とも正確（的中率 100%）。**
- **不正確な座標は 1 件も「確定」になっていない → 黙って誤りを出した件数 = 0。**
- 抽出時刻 ↔ stop_times の内部整合は **100%**（時刻は完全保持）。

### 2.4 内挿アルゴリズムの改善は効果が薄い（検証済み）

「直線内挿」を「OSRM 道路経路に沿った内挿」に変えても改善は小さい。

| 内挿方法 | 中央値誤差 |
|---|---:|
| 直線内挿（現状） | 420 m |
| OSRM 道路経路内挿 | 376 m |

- 改善は約 44 m にとどまり、むしろ悪化する停留所もある。
- 理由: **アンカー（確定座標）が疎で遠く**、OSRM が2アンカー間を結ぶ「最速の車ルート」は**実際のバス経路と異なる**ため。情報不足で内挿では実位置を復元できない。

---

## 3. 考察 — 「80点→100点」の正体

本システムの停留所座標について、正確性は**データの可用性で決まる**：

| 状況 | 得られる精度 | システムの挙動 |
|---|---|---|
| 公式GTFSあり（座標再利用） | ≒ 0 m | 実装済み。正確 |
| P11完全一致・Nominatim命中 | 約 50 m | 正確（確定/要確認） |
| 超ローカル停留所（公式もP11もOSMも無い） | 内挿 約 400 m | **全て要確認**（復元不能） |

- コミュニティバスの停留所座標は**どのオープンデータにも無い**ことが多く、その分は**内挿の推定＝要確認**にせざるを得ない。最終確定は人が地図で行う（本システムの⑤の役割）。
- 重要なのは、**精度が完璧でないこと自体は問題ではない**という点である。官公庁が扱うデータで避けるべきは「不確実なのに正しいと偽って出すこと」であり、本システムは：
  - **正確に出せる所だけ「確定」にする（的中率100%）**、
  - **出せない所は必ず「要確認/未補完」に回す（サイレント誤り0）**。
- すなわち本システムの価値は「完璧な座標」（データ上不可能）ではなく、**「誤りを黙って出さないこと」＋「オープンデータで信頼できる分は自動で埋めること」** にある。これは設計原則〈**正しく失敗**〉の定量的な裏づけである。

---

## 4. 結論

1. 独自手法（P11＋Nominatim＋内挿）の座標精度は**中央値 約232m**。ただしこれは**データ可用性の壁**によるもので、内挿アルゴリズムの改良では埋まらない（OSRM内挿で420→376mに留まる）。
2. **信頼度のキャリブレーションは良好**：確定は的中率100%、不確実は全て要確認/未補完、**サイレント誤り＝0**。
3. 公式GTFSがある路線では座標再利用で**≒0m**を達成しており、実務上は公式データの活用が最有効。
4. したがって本システムの「100点」は、座標を完璧化する（不可能）ことではなく、**手法の適用範囲と限界を厳密に定量化し、"誤りを黙って出さない"ことを保証する**ことで達成される。

---

### 付記：再現方法

```bash
python golden_test.py            # 公式GTFS比較の回帰テスト（PASS/FAIL）
# 個別比較（座標距離メトリクス付き）:
python skills/gtfs-jp-creator/scripts/eval_compare.py \
    --official <公式GTFS.zip> --ours <当方GTFSフォルダ or zip> -o report.md --json report.json
```

合否ゲート（`golden_test.py`）: **サイレント誤り（確定なのに>100m）=0 / 確定の的中率100% / 内部整合(時刻)100%**。
座標の絶対精度・Validatorの ERROR/WARNING は**参考値**として記録する（難路線では独自手法だけだと座標が大きくズレるが、それらは全て要確認へ回るのが正しい挙動）。
