# 利用手順（詳細）— Claude Code で時刻表 → GTFS-JP

README「クイックスタート」の詳細版。**準備（一度だけ）** と **利用の流れ（毎回）** を1枚にまとめる。
初めての人・試用者向け。

# パートA：最初の準備（一度だけ）

## 前提：先に入っているべきもの
| 必要なもの | 用途 | 確認コマンド |
|---|---|---|
| **Git** | repo の取得 | `git --version` |
| **Python 3.10 以上**（pip 付き） | スクリプト実行 | `python --version`（Mac/Linux は `python3 --version`）|
| **Claude Code** | スキルを動かす本体 | （導入済み）|
| **Java 17 以上**（任意） | Step7 の検証だけに必要 | `java -version` |

無い場合：Git は git-scm.com、Python は python.org（インストール時「Add to PATH」にチェック）、
Java は Adoptium(Temurin) など。

## ⚡ 最短セットアップ（コピペ1ブロック）
前提（Git・Python・任意でJava）が入っていれば、**下を丸ごとコピペして実行するだけ**。
取得・依存インストール・Validator jar の自動DL・動作確認まで一括で済む。

**Windows（PowerShell）**
```powershell
git clone https://github.com/k23rs125/gtfs-jp-creator.git
cd gtfs-jp-creator
pip install -r requirements.txt
python skills\gtfs-jp-creator\scripts\download_validator.py        # 検証用jar自動DL（要 Java 17+。不要なら省略可）
python -c "import shapefile, pdfplumber, openpyxl, pykakasi, fitz, pymupdf4llm; print('deps OK')"
```

**Mac / Linux（bash）**
```bash
git clone https://github.com/k23rs125/gtfs-jp-creator.git
cd gtfs-jp-creator
pip3 install -r requirements.txt
python3 skills/gtfs-jp-creator/scripts/download_validator.py      # 検証用jar自動DL（要 Java 17+。不要なら省略可）
python3 -c "import shapefile, pdfplumber, openpyxl, pykakasi, fitz, pymupdf4llm; print('deps OK')"
```
最後に `deps OK` と出れば成功。あとは **このフォルダを Claude Code で開く**だけ。
（環境を分けたい人は `pip install` の前に `python -m venv .venv` → 有効化を挟む。
Linux で `externally-managed-environment` と出る場合も venv を使う。）

以下は上の各行を分解した詳細。トラブル時に参照。

## 手順1：repo を取得
```bash
git clone https://github.com/k23rs125/gtfs-jp-creator.git
cd gtfs-jp-creator
```

## 手順2：Python 依存をインストール
**仮想環境を推奨**（システムを汚さない）。
```bash
python -m venv .venv
# 有効化 ─ Windows(PowerShell): .venv\Scripts\Activate.ps1
#          Mac/Linux:           source .venv/bin/activate
pip install -r requirements.txt
```
入るもの：`pyshp`（P11読込）/ `pdfplumber`（PDF抽出）/ `openpyxl`（Excel）/
`pykakasi`（ふりがな）/ `PyMuPDF` / `pymupdf4llm`。

**確認**（全部 import できれば OK）：
```bash
python -c "import shapefile, pdfplumber, openpyxl, pykakasi, fitz, pymupdf4llm; print('deps OK')"
python skills/gtfs-jp-creator/scripts/run_pipeline.py --help
```

## 手順3：検証ツールの準備（Step7 を使う場合のみ・後回し可）
config に `validate` を付けなければ Step7 はスキップされる。検証する場合のみ：
1. **Java 17+** を入れ、`java -version` が通ることを確認。
2. **jar を自動取得**（最短セットアップに含まれる1コマンド）：
   ```bash
   python skills/gtfs-jp-creator/scripts/download_validator.py
   ```
   最新リリースの CLI jar を `skills/gtfs-jp-creator/tools/gtfs-validator-cli.jar` に配置する。
3. 確認：`java -jar skills/gtfs-jp-creator/tools/gtfs-validator-cli.jar --help`

> 自動取得に失敗する場合のみ手動で：[GTFS Validator リリース](https://github.com/MobilityData/gtfs-validator/releases)
> から `*-cli.jar` をDL → `gtfs-validator-cli.jar` にリネーム → 上記パスに配置。

## 手順4：Claude Code でフォルダを開く
`gtfs-jp-creator` を Claude Code で開くと、`skills/gtfs-jp-creator/SKILL.md`（先頭の frontmatter）が
**スキルとして自動認識**される。利用者がコマンドを覚える必要はない。

## ✅ 手動設定が不要なもの（自動で済む）
- **国土数値情報 P11**（座標補完）→ config に `"p11_prefecture": "福岡県"` と書けば**自動DL**。
- **OSRM**（走行経路）→ 公開デモサーバ。失敗時は直線フォールバックで他Stepは動く。
- **Nominatim**（座標補完の補助）→ 公開API。
- **MinerU**（装飾/画像PDFのOCR）→ 装飾/画像化PDFを扱うときだけ任意で `pip install -U "mineru[core]"`
  （初回 ~3GB のモデルDL。通常のテキストPDF/Excelなら不要）。
  画像PDFは `pdf_to_markdown.py --engine mineru`（既定 **pipeline バックエンド**＝数字に正確で速い）
  → `extract_timetable_markdown.py` で取り込む。

## 動作確認（準備完了チェック）
1. `python -c "import shapefile, pdfplumber, openpyxl, pykakasi, fitz, pymupdf4llm; print('deps OK')"`
2. `python skills/gtfs-jp-creator/scripts/run_pipeline.py --help`
3. （検証使うなら）`java -jar .../gtfs-validator-cli.jar --help`
4. Claude Code でスキルが認識されている

## つまずきやすい点
| 症状 | 対処 |
|---|---|
| `python` が無い/古い | `python3` を使う／3.10+ を入れ直す（PATH追加）|
| `pip install` でビルド失敗 | `python -m pip install -U pip`。PyMuPDF/pyshp は wheel なので通常ビルド不要 |
| 検証で `java not found` | Java 未導入 or PATH 未設定。`java -version` を先に通す |
| 検証で jar が見つからない | 配置先・名前が `skills/gtfs-jp-creator/tools/gtfs-validator-cli.jar` か確認 |
| 文字化け(Windows) | `set PYTHONIOENCODING=utf-8` |
| Mac/Linux | `python`→`python3`、`pip`→`pip3`、パス区切りは `/` |

# パートB：利用の流れ（毎回）

利用者の動きは **3つだけ**：①時刻表を渡す ②条件確認に答える ③結果を確認する。

## 1. 時刻表を渡して頼む
バス時刻表の **PDF / Excel** を Claude Code に添付し、「**これを GTFS-JP にして**」と頼む。

## 2. Claude が抽出（自動）
入力形式に応じて最適経路を選ぶ：テキストPDF=座標方式、Excel=グリッド直読み（往復・上下も自動縦分割）、
Publisher(.pub)=PDF化して座標方式、画像化PDF=OCR(MinerU, 既定pipelineバックエンドで数字に正確)
→ Markdownを `extract_timetable_markdown.py` で取り込み。
「○○線・21停留所・10便（往復）を抽出しました」のように取れたものを提示する。
複数路線（往復が同じ停留所集合）は自動でまとめる。

## 3. ★条件確認（ask-first）
PDF/Excel に**無い**情報は推測せず一括質問する。例：①事業者 ②法人番号 ③運行曜日・運休日
④有効期間 ⑤運賃（**大人/小児/障がい者**の区分別も可）⑥旧/公式GTFSの有無 ⑦行先表示 ⑧対象自治体。

## 4. 利用者が答える（不明でOK）
分かる範囲で。**「不明」と答えてよい**。

## 5. Claude が不明分を調査（要確認つき）
不明な項目だけ Web 調査し暫定値を埋める。必ず「正確とは保証できない（要確認）」と明記。
法人番号は捏造せず公的確認時のみ採用。⑥は「なし」でも BODIK 等で公式GTFSを能動確認。

## 6〜7. 生成→座標補完→検証（自動）
`run_pipeline` で：GTFS-JP生成 → 座標補完の多層（旧/公式feed再利用→P11→同名候補の経路位置選択→
Nominatim→手動→経路ジオメトリ外れ値棄却→内挿推定）→ **座標の信頼度分類**（確定/要確認/未補完。
**第3相＝独立な複数ソースが近接一致したら確定に昇格**）→ 走行経路・多言語・zip化 →
**GTFSビューア(gtfs_viewer.html)生成** → 検証（標準Validator・JP拡張・内部整合・**時刻アノマリ検出**）→
地図 `map_view.html` 生成。

## 8. 結果を確認
標準/JP拡張のERROR数・内部整合 N/N・座標カバレッジ（確定/要確認）・**要確認項目**・
**時刻アノマリ(OCR誤読の疑い)** が報告される。成果物と同じ場所(output_dir)に：
- `gtfs_viewer.html`（路線一覧/時刻表/運賃表/路線図/カレンダー/バス停/データチェックの7タブ。ブラウザで開く）
- `map_view.html`（停留所・経路・便の目視確認）
- `座標_信頼度.csv` / `座標_要確認.csv` / `時刻アノマリ.json`（人の確認用）

> **官公庁提出の核**：要確認の座標・時刻は人が原典照合するまで正式採用しない（＝推測を黙って出さない）。

## 9. 直したいとき
「○○の座標がずれてる」「公式GTFSの座標を使って」「祝日も運休」「○便の時刻が違う」等と言えば、
Claude が config や時刻を直して再生成する（手動座標・`official_feed_url`・`holiday_*` 等）。
