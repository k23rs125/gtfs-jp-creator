## 1. これは何か
- バス時刻表 **PDF / Excel** から **GTFS-JP**（公共交通オープンデータの日本標準）を半自動生成する Claude Code スキル。
- 正確さの源は **決定的スクリプト**。LLM 単体（貼るだけ）では正確なデータにならない。
- 設計思想は「**正しく失敗する**」：不明な値は推測で埋めず、未確定として要確認に回す。

## 2. セットアップ
[**利用手順（詳細）**](利用手順.md)（準備＋利用の流れ＋トラブルシュート）に従うのが確実。要点だけ:
1. `git clone` → `pip install -r requirements.txt`（Python 3.10+）
2. 検証(Step7)を使うなら **Java 17+** と **Validator jar**（`skills/gtfs-jp-creator/tools/` に配置）
3. Claude Code でフォルダを開く（`SKILL.md` が自動認識される）

## 3. 試すこと（どれか）
- **A. 自分の時刻表（Claude Code）**：手元のバス時刻表（テキストPDF / Excel）を渡して「これを GTFS-JP にして」。
  - 抽出 → **条件確認（PDF/Excel に無い項目の一括質問）に答える** → 生成 → 検証 → 地図確認、まで通す。
  - 画像化PDFのときは案内される OCR コマンド（MinerU）でMarkdown化してから取り込む。
- **B. 画面操作アプリ（Streamlit）**：`pip install -r app/requirements.txt` → `streamlit run app/app.py`。
  同梱サンプル（太宰府=往復 / 築城=循環 / こが=画像PDF→OCR）のボタンで一周でき、⑥ビューアまで見られる。
- **C. 同梱サンプルの見本**：`skills/gtfs-jp-creator/references/examples/sample_naogata_eimanji/`（良い出力の見本）。

## 4. 見てほしい観点（チェックリスト）
- [ ] **セットアップ**でつまずいた箇所（手順の不足・エラー・分かりにくさ）
- [ ] **抽出**は時刻表どおりか（停留所・便・時刻の取りこぼし／取り違え。OCR時は誤読がないか）
- [ ] **条件確認**の質問は分かりやすく、過不足なかったか（運賃の区分など）
- [ ] **生成**は通ったか（標準 Validator ERROR 数／JP拡張／内部整合 N/N／時刻アノマリ）
- [ ] **地図**（`map_view.html`）・**ビューア**（`gtfs_viewer.html` の7タブ）で停留所・経路・運賃は妥当か
- [ ] **座標の要確認**（確定/要確認/未補完）と、地図でのピン確定（ドラッグ等）は使いやすいか
- [ ] 「正しく失敗」（不明は埋めず要確認）の方針に納得できたか
- [ ] README / SKILL / ガイドで分かりにくい所
- [ ] 一番良かった点 ／ 一番の改善要望
