# GTFS-JP Creator 使い方・統合ガイド
### 〜 人間向けの導線 ＋ ツール付きLLMの司令書 を1枚に 〜

このガイドは **バス時刻表（PDF / Excel）から GTFS-JP データを半自動生成する** ための単一の手引きである。
2通りの読み方ができる。

- **(A) 人間として手順を追う** … 第1部を読む。何ができ、何を準備し、どう実行するか。
- **(B) ツール付きLLMに実行させる** … 本ガイドごと Claude Code 等（スクリプトを実行できるLLM）に渡す。LLMは第2部の司令書に従う。

---

## 設計の核心（最初に必ず読む）

**LLMは「構造化（Step2）」と「司令役」だけを担う。抽出・各ファイル生成・座標補完・走行経路・検証は、すべて決定的なスクリプトが行う。**

> ⚠️ **「PDFをLLMに貼るだけで全部作る」方式は採用しない。**
> LLM単体に丸投げすると、長い表の末尾省略・時刻の取り違え・**座標の捏造**・ファイル間の相互参照崩れが起き、信頼できる GTFS にならない。
> 量が多く正確さが要る部分は決定的処理に任せ、LLMは「紙面の解釈（路線・方向・循環の判断）」と「手順の進行・利用者への質問」に限定する。これが本ツールの再現性・誤差ゼロの源である。

決まらない値は **推測で埋めず**、利用者に質問する（「正しく失敗」）。

# 第1部：人間向け — 何ができて、何が要るか

## 1.1 入力と出力
- **入力**：バス時刻表（テキストPDF / Excel(.xlsx) / 装飾PDF / 画像PDF）
- **出力**：GTFS-JP 一式（zip。JP拡張 agency_jp / routes_jp 等を含む）＋ 検証レポート ＋ 地図ビュー(HTML)

## 1.2 必要なもの（環境）
| 用途 | 必要物 | 備考 |
|---|---|---|
| 抽出・生成・座標補完（主経路） | Python ＋ `pip install -r requirements.txt` | pdfplumber / pyshp / requests / pykakasi |
| 座標補完 P11 | 国土数値情報 P11（**対象都道府県**の SHP） | `download_p11.py <都道府県>` で取得 |
| 走行経路 shapes | OSRM エンドポイント（ネット） | `/route` を利用 |
| 検証 | Java 17 以上 ＋ gtfs-validator-cli.jar | `tools/` に配置 |
| 装飾PDFの抽出（任意） | MinerU（GPU推奨） | CPUは30〜60分。テキストPDFなら不要 |

## 1.3 全体の流れ（LLMを使うのは Step2 だけ）
```
[Step1] 時刻表(PDF/Excel)  → 抽出（座標方式 / Excel直接 / OCR）         ※決定的
[Step2] 抽出結果           → 構造化中間表現 JSON                        ※LLM（路線・方向・循環の判断）
[条件確認] JSON            → 要入力サマリを提示し、利用者が不足項目を補完  ※質問は一度に集約
[Step3]  JSON             → GTFS-JP 各ファイル（JP拡張含む）             ※決定的
[Step3.5] stops           → 緯度経度補完（旧feed→P11→Nominatim→手動）   ※決定的・手動最優先
[Step4]  stop_times       → shapes.txt（OSRM routing）                  ※決定的
[Step6]  stops/routes     → translations.txt（ふりがな pykakasi / 英訳） ※決定的（一部LLM）
[Step5]  全ファイル        → zip パッケージング                          ※決定的
[Step7]  zip              → 検証（MobilityData ＋ JP拡張）＋内部整合検証   ※決定的
```

## 1.4 利用者が決めること（PDFに無い項目は質問される）
時刻表に載っていない項目は推測されず、**条件確認サマリ**でまとめて尋ねられる。
- 事業者情報（正式名称・所在地・電話・URL・連絡先メール）
- 運行する曜日 ／ フィードの有効期間
- 自動補完できない座標（同名あいまい・接尾辞付き名称など）
- 路線名・方向・循環の割り当て（紙面の解釈が要る点）
- 要予約・特殊便の扱い

## 1.5 最短手順（テキストPDFの例・4コマンド）
```powershell
# 1. 抽出（座標方式・該当ページ指定。2方面は見出し行で自動縦分割）
python skills\gtfs-jp-creator\scripts\extract_timetable_coords.py "時刻表.pdf" -o test_demo\extract.json --page 2

# 2. 構造化（LLMが extract.json → claude.json を作成。条件確認で不足項目を補完）

# 3. 生成〜検証を一括（config を指定）
python skills\gtfs-jp-creator\scripts\run_pipeline.py --config test_demo\pipeline_config.json

# 4. 内部整合検証（抽出した時刻が stop_times まで保たれているか）
python skills\gtfs-jp-creator\scripts\verify_stop_times_vs_extract.py test_demo\extract.json --gtfs test_demo\out\gtfs
```

# 第2部：LLMオーケストレーション司令書

> **あなた（LLM）への指示**
> あなたの役割は **(1) 構造化（Step2）** と **(2) 手順の進行・利用者への質問** である。
> 抽出・生成・座標補完・shapes・検証は **下記スクリプトを実行**して行い、結果を解釈する。
> 決まらない値は推測せず、条件確認サマリで利用者にまとめて質問する。各段で品質ゲートを必ず確認する。
> Windows では cp932 クラッシュ回避のため `PYTHONIOENCODING=utf-8` を付けて実行する。

## 起動手順（このリポジトリURLとPDFを渡されたAI向け）
**前提：あなたはスクリプトを実行できる（shell＋ネットがある）こと。** そうでなければ正確に作れないので、利用者に「コードを実行できる環境（Claude Code 等）が必要」と伝えて止まる。

> ⚠️ **スクリプトを必ず実行する。読んで要約したり、出力を“それらしく”捏造（模倣）したりしない。** これは決定的処理に任せることが正確さの条件だからである。

URLとPDFを渡されたら、まず環境を自走で整える：
```bash
# 1. 取得と依存インストール
git clone https://github.com/k23rs125/gtfs-jp-creator
cd gtfs-jp-creator
pip install -r skills/gtfs-jp-creator/requirements.txt

# 2. 座標補完用 P11 を「対象地域の都道府県」で取得（例：福岡県）
python skills/gtfs-jp-creator/scripts/download_p11.py 福岡県 --year 22

# 3. （検証する場合）gtfs-validator-cli.jar を tools/ に配置（Java 17 以上が必要）
```
以降は本ガイドの **S1 → S2（条件確認で利用者に質問）→ S3 → S4（品質ゲート）→ S5** を順に実行する。

## S0. 前提確認
- リポジトリ位置・Python環境・**対象地域の都道府県**（P11取得に必須）を確認する。

## S1. 抽出（決定的・スクリプト実行）
入力形式で経路を選ぶ。
- **テキストPDF**：`extract_timetable_coords.py <pdf> -o extract.json [--page N]`
  - ページ未指定なら時刻トークン最多のページを自動選択。`--page` で明示すると確実。
  - 弓削線のような **同一時刻列の2方面** は、括弧付き方面見出し行で自動的に縦分割される。
  - 画像化PDF（文字なし）は警告し OCR 経路へ誘導される。
- **Excel**：`extract_timetable_excel.py <xlsx> -o extract.json [--sheet 名 --name-col A]`
- **装飾PDF**：MinerU → Markdown → あなたが構造化。
- 出力 `extract.json` の **`needs_confirmation` を必ず読む**（時刻逆行＝要予約寄り道／番号なし候補＝案内文混入 など）。

抽出JSON（座標方式）の便は `cells:[{seq,num,name,time,reserve}, ...]` を持つ。**num＝停留所番号、reserve＝要予約フラグ** が使える。

## S2. 構造化（あなたの仕事）＋ 条件確認
`extract.json` ＋ 利用者回答 → **構造化中間 JSON** を作る。スキーマ：
```
agency, agency_jp, office_jp, routes[], stops[], trips[], stop_times[],
calendar[], calendar_dates[], fare_attributes[], fare_rules[], _meta
```
構造化の鉄則：
1. **停留所は「名称」で区別する。** 同じ番号でも名称が違えば別の物理停留所（例「○○前（支所側）」と「（銀行側）」＝道路の反対側）。番号だけで統合しない。
2. **要予約バス停（reserve=True）は既定で除外**（利用者が残すと言えば別）。
3. **案内文の混入**（「お問い合わせ」欄が名前列に入る等。`needs_confirmation` に出る）は停留所にしない。
4. **便名・方向(direction_id)・循環** は座標から決まらない。原典と照合し、利用者承認のうえ割り当てる。
5. 時刻は `HH:MM:SS`、`arrival=departure`。stop_sequence は1始まり。

**条件確認サマリ**を提示し、🟦自動検出 / 🟨自動補完 / 🟧要入力 に分類。🟧（PDF外メタ：事業者・曜日・有効期間・連絡先・自動補完不能な座標）を **一度にまとめて質問**する。逐次質問しない。**推測で埋めない。**

## S3. 生成〜検証（決定的・run_pipeline.py）
config JSON を作る：
```json
{
  "feed_name": "...", "input_json": "test_demo/claude.json",
  "output_dir": "test_demo/out", "context": "○○県○○市",
  "p11_shapefile": "p11_data/P11-22_40_SHP/P11-22_40_SHP/P11-22_40.shp",
  "use_nominatim": true,
  "manual_coords": "test_demo/manual_coords.json",
  "validate": true
}
```
- ⚠️ **P11 は対象地域の都道府県 SHP を使う。** 無ければ `download_p11.py <都道府県> --year 22` で取得。
  **県を取り違えると、全国に同名のあるバス停が遠方に誤マッチする**（例：福岡=40 を 沖縄=47 と取り違え → 「高良」が沖縄にマッチ）。
- `--dry-run` で計画確認 → 本実行。座標補完は 旧feed→P11→Nominatim→**手動(最優先)** の順。

## S4. 品質ゲート（必須・省略不可）
1. **内部整合検証**：`verify_stop_times_vs_extract.py extract.json --gtfs out/gtfs`
   → `[PASS] N/N=100%` を確認（抽出時刻が stop_times まで保たれている証拠）。
2. **座標の外れ値チェック**：全停留所座標の中心から大きく外れる点を探す。
   医院・寺・店名など **全国に同名がある停留所** は誤マッチしやすい（例：長福寺が2自治体に存在）。
   接尾辞付き名称（「○○前（支所側）」）は P11 照合を外すので **手動座標(by_stop_name)** で補う。
3. **検証 ERROR 0**：MobilityData と JP拡張がともに ERROR 0 か。残る WARNING の意味を利用者に説明する
   （例：`missing_feed_contact_email_and_url` は事業者未定のため／`stop_too_far_from_shape` はコミュニティバスで発生しうる軟警告）。

## S5. 成果物の集約
`output_dir`（feedフォルダ）に **zip・gtfs/・validation/・stoptimes_verify.*・map_view.html** を揃える。

# 付録A：条件確認で必ず聞くこと（チェックリスト）
- [ ] agency：事業者名／GTFSのagencyに自治体・運行事業者どちらを使うか
- [ ] agency_jp：正式名称・郵便番号・所在地・URL・連絡先メール
- [ ] 運行する曜日（calendar）／運休日（お盆・年末年始等は calendar_dates）
- [ ] 有効期間（feed_start / feed_end）
- [ ] 路線名・方向・循環の割り当て
- [ ] 要予約・特殊便の扱い
- [ ] 自動補完できなかった座標、同名あいまいの選択

# 付録B：よくある落とし穴（実例）
| 症状 | 原因 | 対処 |
|---|---|---|
| 一部の座標が遠方（県外）に付く | P11の**県取り違え** | 対象県のP11を `download_p11.py` で取得し直す |
| 同名停留所が誤った場所に付く | 全国同名（例：長福寺＝筑前町/久留米市） | 外れ値検知 → 近い方を手動座標で採用 |
| 接尾辞付き停留所が未補完 | 「（支所側）」等でP11照合を外す | 手動座標(by_stop_name)で補う |
| 同番号の停留所が1つに統合される | 番号で統合した | **名称で区別**（反対側は別停留所） |
| 時刻が逆行/断片便 | 要予約バス停への寄り道 | 要予約を除外（needs_confirmationに出る） |
| 表に無い「停留所」が出る | 案内文(お問い合わせ欄等)の混入 | needs_confirmationを確認し除外 |
