# GTFS-JP 半自動生成アプリ（Streamlit MVP）

バス時刻表（PDF / Excel）をアップロードすると、**抽出 → Claudeで構造化 → 条件確認フォーム →
生成・座標補完・検証 → 地図プレビュー・GTFS-JP ダウンロード** までを画面上で行う半自動アプリ。

> 設計：正確さの源は決定的スクリプト。LLM(Claude API) は **構造化(Step2) の判断だけ**。
> PDF/Excel に無いメタ情報（事業者・運行日・運賃など）は推測せず **条件確認フォームで人が入力**する
> （＝「正しく失敗」）。スキル本体（`skills/gtfs-jp-creator/scripts/`）をエンジンとして再利用している。

## 必要なもの
- スキル本体のセットアップ（リポジトリ README のクイックスタート）が済んでいること
  （Python 依存・任意で Java＋Validator jar）。
- 追加で `pip install streamlit anthropic streamlit-folium folium`（`app/requirements.txt` でも可）。
- 構造化を自動化するなら **ANTHROPIC_API_KEY**（環境変数 or 画面で入力）。
  キーが無くても、画面の decision-spec 欄に貼り付け／編集すれば動く。
- `apply_decisions.py`（リポジトリ直下・構造化の決定的展開に使用）。

## 起動
```bash
pip install -r app/requirements.txt   # streamlit / anthropic / streamlit-folium / folium
# 画像化PDFのOCRを使うなら（任意・重い）: pip install -U "mineru[core]"
streamlit run app/app.py
```
ブラウザで `http://localhost:8501` が開く（初回のメール入力は空Enterでよい）。

## 入力できる時刻表
- **Excel(.xlsx)** / **テキストPDF**：そのまま「抽出する」。
- **画像化PDF（スキャン）**：文字が無いので直接は不可。①で OCR コマンドが案内される
  （`pdf_to_markdown.py --engine mineru`、既定 pipeline バックエンドで数分・数字に正確）。
  出来た **`.md` を①に再アップロード**すると取り込める。
- サンプルボタン同梱：太宰府(往復xlsx) / 築城(循環・変則便xlsx) / こが(画像PDF→OCR md)。

## 画面の流れ
1. **① アップロード**：`.xlsx` / テキストPDF / OCR後の `.md` を選び「抽出する」（サンプルも可）。
2. **② 路線の割り当て**：停留所の並びが近いブロックを自動で同一路線にまとめ（多路線対応・往復は
   方向0/1）、表で路線名・方向を編集できる。**平日/土日で時刻が違う**路線は「運行日」列を変えると
   別カレンダー（複数ダイヤ）で出力。詳細欄で Claude 構造化や decision-spec 手編集も可。
3. **自動確認の結果**：方向・循環・便/停留所など「時刻表から読み取れたこと」を提示。PDF/Excel に
   無い情報は推測せず③で質問する。
4. **⏰ 時刻表の確認・修正**：全便×全停留所の時刻を表で表示。原典と見比べて直接編集できる
   （空欄＝通過）。OCR誤読の疑いセルはヒント併記（便間パターン＋逆行で検出）。
5. **③ 条件確認**：路線名・事業者・法人番号・運行曜日・有効期間・対象自治体に加え、
   **運賃（大人/小児/障がい者の区分別、または区間運賃CSV）**・**運休日（祝日/年末年始/お盆）**を入力
   （不明は空欄でOK。ただし路線名は必須＝空だと赤エラーで停止）。フォーム送信で一括反映。
6. **生成**：`apply_decisions` → `run_pipeline`（生成・座標補完・検証・ビューア生成）。
7. **④ 結果**：内部整合・**Validator ERROR の内容と該当便**・座標カバレッジ・地図プレビュー・
   要確認リスト・**GTFS-JP(zip) ダウンロード**。
8. **⑤ 座標の確認（地図）**：確定=緑／要確認=橙／未補完=赤。**ピンをドラッグ→クリックで位置確定**、
   地図クリックや座標入力でも確定できる。**全部が確定になるまで「公式提出可」にしない**
   （＝推測座標を人が確認するまで正式採用しない）。官公庁提出向けの核。
9. **⑥ GTFSビューア**：作成した GTFS を単一HTMLで閲覧（📋路線一覧/🕐時刻表/💴運賃表/🗺️路線図/
   📅運行カレンダー/🚏バス停一覧/✓データチェック）。DLしてブラウザで開ける。

## 正確さの工夫（官公庁提出向け）
- **路線名ガード**：空のまま生成しない（Validator ERROR 防止）。
- **時刻アノマリ検出**：OCR等の数字読み違いを便間パターンで検出（自動修正はしない）。
- **座標信頼度＋第3相**：単一ソースは要確認、独立な複数ソース（公式feed/P11/Nominatim）が
  近接一致したら確定に昇格。**要確認は人が地図確認するまで「公式提出可」にしない**。

## 制限（MVP）
- 画像化PDFは OCR(MinerU) を**アプリ外で実行**してから `.md` を取り込む運用（CPUで数分）。
- 完全自動ではない（条件確認・座標確認の人手は設計上残す＝正しく失敗）。
- 座標第3相は複数ソース使用時に効く（`use_nominatim` ON か 公式feed指定）。Nominatim は遅い。
